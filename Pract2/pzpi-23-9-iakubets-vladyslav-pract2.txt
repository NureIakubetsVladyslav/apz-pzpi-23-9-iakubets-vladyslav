3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про систему
Назва системи: Discord
Розробник: компанія Discord Inc.
Discord є сучасною комунікаційною платформою, призначеною для обміну повідомленнями, голосового та відеозв’язку між користувачами у режимі реального часу. Система орієнтована як на геймерів, так і на широке коло користувачів, включаючи навчальні спільноти, робочі команди та онлайн-спільноти за інтересами.
Платформа забезпечує можливість створення серверів (спільнот), у межах яких користувачі можуть організовувати текстові та голосові канали для спілкування. Основними функціями системи є обмін текстовими повідомленнями, передача мультимедійного контенту, здійснення голосових та відеодзвінків, а також інтеграція з іншими сервісами.
Система працює за клієнт-серверною моделлю та доступна у вигляді вебзастосунку, десктопної та мобільної версій, що забезпечує зручний доступ з різних пристроїв. Discord активно використовується для організації онлайн-спілкування, координації дій користувачів та підтримки постійного зв’язку в межах спільнот.
3.2 Архітектурний стиль системи
Архітектура платформи Discord базується на поєднанні мікросервісного та клієнт-серверного підходів із використанням подієво-орієнтованої взаємодії компонентів.
Основою системи є клієнт-серверна архітектура, у якій клієнтські застосунки (веб, десктоп та мобільні) взаємодіють із серверною частиною через API. Це забезпечує централізовану обробку запитів, керування користувачами та зберігання даних. Для забезпечення масштабованості платформа використовує мікросервісну архітектуру, у межах якої функціональність розподілена між окремими сервісами (обробка повідомлень, керування користувачами, голосовий зв’язок тощо), що дозволяє незалежно масштабувати компоненти та підвищує відмовостійкість.
Важливою особливістю є використання подієво-орієнтованої архітектури, яка забезпечує обробку подій у реальному часі. Передача повідомлень та інші дії реалізуються через асинхронний обмін даними між сервісами. Для цього застосовуються технології, зокрема WebSocket, що дозволяють підтримувати постійне з’єднання між клієнтом і сервером.
Поєднання зазначених архітектурних підходів забезпечує високу продуктивність, масштабованість та стабільність роботи системи навіть при значному навантаженні.
3.3 Основні компоненти системи
Архітектура платформи Discord складається з кількох основних компонентів, кожен з яких виконує окрему функцію в системі.
Інтерфейс користувача представлений вебзастосунком, десктопною та мобільною версіями. Вебклієнт реалізований із використанням сучасних JavaScript-технологій, а десктопна версія побудована на базі Electron, що дозволяє використовувати єдину кодову базу. Інтерфейс забезпечує відображення повідомлень, взаємодію з каналами та підтримку голосових і відеофункцій.
Серверна частина відповідає за обробку HTTP-запитів та постійних з’єднань від клієнтів. Взаємодія здійснюється через REST API та протокол WebSocket, що дозволяє реалізувати обмін даними у реальному часі. Сервер також забезпечує автентифікацію користувачів та керування сесіями.
Рівень бізнес-логіки реалізований у вигляді мікросервісів, що обробляють основні функції системи: обмін повідомленнями, управління серверами, обробку подій. Для взаємодії між сервісами використовуються асинхронні механізми та черги повідомлень, що підвищує продуктивність і масштабованість.
Рівень зберігання даних включає використання баз даних для довготривалого збереження інформації (користувачі, повідомлення, налаштування), а також систем кешування для швидкого доступу до часто використовуваних даних. Це дозволяє зменшити навантаження на основні бази даних та підвищити швидкість роботи системи.
Допоміжні сервіси забезпечують обробку медіаконтенту, голосового зв’язку (VoIP), систему сповіщень та інтеграцію з іншими сервісами. Вони реалізуються як окремі компоненти, що можуть масштабуватися незалежно від основної системи.
3.4 Взаємодія компонентів
Взаємодія компонентів платформи Discord реалізована на основі клієнт-серверної моделі з використанням асинхронного обміну даними та подієво-орієнтованого підходу.
Клієнтські застосунки (веб, десктоп та мобільні) взаємодіють із серверною частиною через HTTP-запити (REST API) та постійні WebSocket-з’єднання. HTTP використовується для виконання стандартних операцій, таких як авторизація, отримання даних або надсилання повідомлень, тоді як WebSocket забезпечує миттєву доставку подій у реальному часі.
При обробці повідомлень клієнт надсилає запит до API Gateway, який передає його відповідному мікросервісу. Сервіс повідомлень здійснює обробку даних, зберігає повідомлення у базі даних та генерує подію про створення нового повідомлення. Дана подія потрапляє до черги подій, після чого обробляється Gateway WebSocket для доставки іншим користувачам.
Асинхронна взаємодія між компонентами дозволяє зменшити затримки, підвищити продуктивність системи та забезпечити стабільну роботу при великій кількості одночасних користувачів. Використання окремих сервісів для голосового зв’язку та сповіщень дозволяє незалежно обробляти різні типи навантаження.
 
Рисунок 3.1 – Компонентна архітектура платформи Discord
На рисунку 3.1 зображено загальну структуру взаємодії компонентів платформи Discord. Клієнтські застосунки взаємодіють із серверною частиною через API Gateway та WebSocket Gateway. Основна функціональність реалізована у вигляді мікросервісів, які взаємодіють із рівнем зберігання даних та допоміжними компонентами, такими як кеш і черга подій.
 
Рисунок 3.2 – Послідовність обробки та доставки повідомлення в Discord
На рисунку 3.2 показано процес обробки повідомлення в системі. Користувач надсилає повідомлення через клієнтський застосунок, після чого воно передається через API Gateway до сервісу повідомлень. Сервіс зберігає повідомлення у базі даних та формує подію, яка через чергу подій передається до WebSocket Gateway. Далі повідомлення доставляється іншим користувачам каналу у режимі реального часу.
3.5 Масштабованість та надійність
Платформа Discord розрахована на обслуговування великої кількості користувачів одночасно, тому при її проєктуванні особлива увага приділяється масштабованості та надійності системи.
Масштабованість системи забезпечується за рахунок використання мікросервісної архітектури, що дозволяє незалежно масштабувати окремі компоненти залежно від навантаження. Наприклад, сервіси обробки повідомлень або WebSocket Gateway можуть масштабуватися горизонтально шляхом додавання нових екземплярів.
Для розподілу навантаження використовується балансування трафіку між серверами, що дозволяє рівномірно обробляти запити користувачів та уникати перевантаження окремих вузлів. Це особливо важливо для системи реального часу, де затримки повинні бути мінімальними.
Надійність системи досягається за рахунок використання відмовостійких підходів, таких як реплікація даних та дублювання сервісів. У випадку відмови одного з компонентів його функції можуть бути швидко передані іншому екземпляру, що забезпечує безперервність роботи системи.
Для підвищення продуктивності застосовується кешування, що дозволяє зменшити кількість звернень до бази даних та прискорити обробку запитів. Крім того, використання черг подій дозволяє обробляти навантаження асинхронно та уникати блокування основних процесів системи.
Важливу роль відіграє використання хмарної інфраструктури та контейнеризації, що спрощує розгортання, масштабування та управління сервісами. Це дозволяє системі швидко адаптуватися до змін навантаження та забезпечувати стабільну роботу навіть при пікових навантаженнях.

3.6 Приклади програмної реалізації
Для демонстрації архітектурних принципів платформи Discord розглянемо приклад серверної обробки повідомлень у клієнт-серверній системі. Даний приклад ілюструє взаємодію між клієнтом, серверною логікою та механізмом асинхронної доставки повідомлень у реальному часі.
Нижче наведено спрощений приклад реалізації API-контролера на платформі ASP.NET Core, який відповідає за створення повідомлення, його збереження та подальшу доставку іншим користувачам:
1  using Microsoft.AspNetCore.Mvc;
2  using System;
3  using System.Collections.Generic;
4  using System.Runtime.InteropServices;
5  using System.Threading.Tasks;
6  
7  [ApiController]
8  [Route("api/messages")]
9  public class MessageController : ControllerBase
10 {
11     private static readonly List<Message> _messages = new();
12     private readonly IWebSocketNotifier _notifier;
13 
14     public MessageController(IWebSocketNotifier notifier)
15     {
16         _notifier = notifier;
17     }
18 
19     [HttpPost]
20     public async Task<IActionResult> SendMessage([FromBody] CreateMessageDto dto)
21     {
22         if (string.IsNullOrWhiteSpace(dto.Content))
23             return BadRequest("Message cannot be empty");
24 
25         var message = new Message
26         {
27             Id = Guid.NewGuid(),
28             ChannelId = dto.ChannelId,
29             Content = dto.Content,
30             CreatedAt = DateTime.UtcNow
31         };
32 
33         _messages.Add(message);
34 
35         await _notifier.NotifyChannelAsync(dto.ChannelId, message);
36 
37         return Ok(message);
38     }
39 }
40 
41 public class CreateMessageDto
42 {
43     public string ChannelId { get; set; }
44     public string Content { get; set; }
45 }
46 
47 public class Message
48 {
49     public Guid Id { get; set; }
50     public string ChannelId { get; set; }
51     public string Content { get; set; }
52     public DateTime CreatedAt { get; set; }
53 }
54 
55 public interface IWebSocketNotifier
56 {
57     Task NotifyChannelAsync(string channelId, Message message);
58 }
Після отримання HTTP-запиту від клієнта контролер виконує перевірку вхідних даних, створює об’єкт повідомлення та зберігає його у сховищі (у прикладі використано спрощену імітацію бази даних). Далі викликається механізм сповіщення, який імітує подієво-орієнтовану доставку повідомлення іншим користувачам через WebSocket.
Такий підхід демонструє розділення відповідальностей між компонентами системи: контролер відповідає за обробку запитів, модель - за структуру даних, а окремий сервіс (WebSocket-нотифікатор) - за доставку повідомлень. Використання асинхронних викликів дозволяє підвищити продуктивність системи та забезпечити обробку великої кількості запитів одночасно.
Наведений приклад відображає ключові принципи побудови сучасних розподілених систем, зокрема клієнт-серверну взаємодію, подієво-орієнтовану архітектуру та підтримку обміну даними у реальному часі.
3.7 Переваги архітектурних рішень Discord 
Архітектурні рішення, використані у платформі Discord, забезпечують ефективну роботу системи при високому навантаженні та великій кількості користувачів.
Основні переваги:
1.	Мікросервісна архітектура - дозволяє розділити систему на незалежні компоненти, що спрощує масштабування, розробку та підвищує відмовостійкість.
2.	Подієво-орієнтований підхід - забезпечує асинхронну обробку подій та швидку реакцію системи у реальному часі.
3.	Використання WebSocket - дозволяє підтримувати постійне з’єднання між клієнтом і сервером для миттєвої доставки повідомлень.
4.	Горизонтальне масштабування - забезпечує можливість обробки великої кількості користувачів шляхом додавання нових серверів.
5.	Кешування та черги подій - підвищують продуктивність системи та зменшують навантаження на базу даних.
6.	Розділення на рівні - спрощує підтримку, розвиток системи та впровадження нових функцій.

Запит В.3 до ChatGPT. Тема: приклад програмної реалізації.
Напиши приклад коду мовою C# для ASP.NET Core, який демонструє обробку повідомлення в системі, схожій на Discord. Код має показувати отримання повідомлення через API, його збереження та асинхронну доставку через WebSocket-нотифікатор.
1  using Microsoft.AspNetCore.Mvc;
2  using System;
3  using System.Collections.Generic;
4  using System.Runtime.InteropServices;
5  using System.Threading.Tasks;
6  
7  [ApiController]
8  [Route("api/messages")]
9  public class MessageController : ControllerBase
10 {
11     private static readonly List<Message> _messages = new();
12     private readonly IWebSocketNotifier _notifier;
13 
14     public MessageController(IWebSocketNotifier notifier)
15     {
16         _notifier = notifier;
17     }
18 
19     [HttpPost]
20     public async Task<IActionResult> SendMessage([FromBody] CreateMessageDto dto)
21     {
22         if (string.IsNullOrWhiteSpace(dto.Content))
23             return BadRequest("Message cannot be empty");
24 
25         var message = new Message
26         {
27             Id = Guid.NewGuid(),
28             ChannelId = dto.ChannelId,
29             Content = dto.Content,
30             CreatedAt = DateTime.UtcNow
31         };
32 
33         _messages.Add(message);
34 
35         await _notifier.NotifyChannelAsync(dto.ChannelId, message);
36 
37         return Ok(message);
38     }
39 }
40 
41 public class CreateMessageDto
42 {
43     public string ChannelId { get; set; }
44     public string Content { get; set; }
45 }

49     public Guid Id { get; set; }

53 }
